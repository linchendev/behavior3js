## Context

当前仓库的核心价值在于一套稳定的行为树运行时语义，而不是 JavaScript 这一门语言本身。`BehaviorTree.tick`、`BaseNode._execute`、`Blackboard` 作用域以及 `load`/`dump` 的 JSON 结构共同构成了外部兼容面。用户已经明确要求 Go 版本保持现有配置文件可直接使用，并尽量保留目录组织、类型名、变量名和函数名，方便与现有源码逐文件对照。

仓库目前仍以 JavaScript 版本为主，因此 Go 迁移必须是增量的、非破坏性的。新实现不能强迫现有用户迁移，也不能为了 Go 风格重写掉原有概念模型，否则会削弱兼容目标。用户进一步要求将 Go 实现完整收敛到 `behavior3go/` 目录下，使其在仓库内表现为与现有 JS 项目并列的独立项目，而不是在原项目源码树中混放语言实现。

## Goals / Non-Goals

**Goals:**
- 新增一套位于 `behavior3go/` 的 Go 版核心库，行为上对齐现有 `src/` 下的运行时。
- 保持 JSON 树配置、节点命名和加载流程兼容，不要求用户改写配置文件。
- 在仓库层面保持高可对照性，`behavior3go/` 内部目录尽量镜像现有 `src/` 结构，例如 `behavior3go/core`、`behavior3go/actions`、`behavior3go/composites`、`behavior3go/decorators`。
- 通过内部实现优化提升可维护性和执行效率，但不改变外部可观察行为。
- 用测试和共享夹具验证 JS 与 Go 结果一致。

**Non-Goals:**
- 不在本次变更中移除、替换或停止维护现有 JavaScript 实现。
- 不引入新的配置格式，不要求将 JSON 改写为 YAML、TOML 或 Go 原生结构。
- 不为了迎合 Go 习惯而修改现有节点语义、状态常量含义或加载顺序。
- 不覆盖 UI 编辑器、文档站点主题或其他非核心运行时代码。

## Decisions

### 1. 使用 `behavior3go/` 独立项目，而不是覆盖现有 JS 代码
决策：在仓库根目录新增 `behavior3go/` 子树，并将 Go 代码、`go.mod`、测试与文档全部收敛在该目录下；`behavior3go/` 内部再按 `src/` 的职责划分镜像目录。

原因：这满足“非破坏性优化”和“代码层面对照”的目标，同时明确了 JS 与 Go 是两个独立项目，避免构建脚本、发布元数据和测试入口互相污染。相比直接在根目录引入 Go 包或重排现有文件，`behavior3go/` 独立项目更利于逐文件比对和分阶段迁移。

备选方案：
- 直接在根目录引入 Go 包：会让 JS 与 Go 的边界不清晰，给现有 Node 工具链带来噪音。
- 单独新仓库：兼容性验证和源码对照成本更高。

### 2. 保持核心类型名和节点名与 JS 一致
决策：Go 实现保持 `BehaviorTree`、`Blackboard`、`Tick`、`BaseNode`、`Sequence`、`Repeater` 等名称不变；字段和方法在语言允许范围内尽量保留原命名。

原因：兼容迁移的难点不在算法，而在读者认知切换。名称一致能让开发者直接从 JS 源码映射到 Go 代码。对于 Go 中必须导出的标识符，优先保持词形一致，例如 `Load`、`Dump`、`Tick`；无法逐字保留的私有辅助方法，在注释中标明对应 JS 名称。

备选方案：
- 完全采用 Go idiomatic 命名：更“地道”，但会显著增加比对和审查成本。

### 3. 复用现有 JSON 契约，加载流程按 JS 语义实现
决策：Go 版本的 `Load`/`Dump` 使用与 JS 一致的数据模型：树级字段、`nodes` 字典、`root`、`properties`、`custom_nodes` 等结构保持不变；加载顺序继续采用“先建节点表，再连接关系，最后设置根节点”。

原因：这直接满足现有配置兼容要求，也能保证与当前 `BehaviorTree.load` 的错误路径一致，例如未知节点名立即报错。

备选方案：
- 先映射为 Go 特有结构再回写：可读性更好，但更容易偏离既有 JSON 契约。

### 4. 用注册表机制承接自定义节点
决策：内置节点通过固定注册表解析，自定义节点通过名称到构造函数的注册表注入，语义对齐 JS 的 `names` 参数。

原因：这样既保留现有“按名字解析节点”的加载方式，也避免在 Go 里滥用反射。注册表是 Go 中最稳定、可测试、可静态分析的等价实现。

备选方案：
- 反射创建节点：灵活，但可维护性和错误信息都较差。

### 5. 优化仅限内部实现，不改变观察结果
决策：允许在 Go 实现中使用预分配 slice、显式错误类型、共享夹具测试和基准测试，但任何优化都必须以“相同输入得到相同状态转移、黑板状态和 dump 结果”为前提。

原因：用户要求“非破坏性的优化”，因此性能和工程性优化必须从属于兼容性。

## Risks / Trade-offs

- [命名与 Go 习惯冲突] → 通过“公开 API 保持一致，私有细节适度 Go 化”的方式平衡一致性与可维护性。
- [JSON 兼容只做到结构相似而非语义相同] → 为关键夹具建立 JS/Go 对照测试，覆盖 `load`、`tick`、`dump`、错误路径和黑板作用域。
- [并行维护两个独立项目增加成本] → 将 JS 版本作为语义基线，`behavior3go/` 通过镜像结构和共享测试样例降低维护负担。
- [自定义节点扩展点在 Go 中过度抽象] → 先只支持与现有 `names` 机制等价的注册表，不提前设计额外插件系统。

## Migration Plan

1. 在 `behavior3go/` 下建立独立项目骨架、`go.mod` 和镜像目录。
2. 先迁移常量、基础类型和 `Blackboard`，再迁移 `BaseNode`、`Tick`、`BehaviorTree`。
3. 在 `behavior3go/` 内迁移内置 `actions`、`composites`、`decorators`，保持文件名和类型名可对照。
4. 为现有 JSON 示例和核心测试场景建立共享夹具，补齐 `behavior3go/` 侧单元测试。
5. 补充 README 或迁移文档，说明 `behavior3js` 与 `behavior3go` 的关系、兼容范围和限制。

回滚策略：`behavior3go/` 独立于现有 JS 发布链路，若兼容性未达到要求，可停止发布 Go 模块而不影响现有 JavaScript 用户。

## Open Questions

- Go 版本是否需要在首批实现中同时覆盖 `Condition` 抽象与所有内置节点，还是允许先完成核心运行时与最常用节点。
- 是否需要提供一个与 JS `index.js` 对应的聚合入口包，还是优先通过子包导出保持目录对照。
- 兼容性验证是否需要引入 golden 文件形式的 `dump` 结果快照，以便后续持续回归。
